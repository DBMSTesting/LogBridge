- Title: FileReaderManager leaks file streams after compaction (deleted files held open)
- Symptom: After compactions complete, the deleted source TsFiles still hold open file descriptors; `lsof` shows growing list of (deleted) entries; may eventually hit 'too many open files'
- Root cause: FileReaderManager caches readers per file; on compaction the source file is deleted but the cached reader is not invalidated/closed
- Fix:
      Upgrade to 1.3.x+ — fixed by PR https://github.com/apache/iotdb/pull/10688
      Workaround: restart DataNode periodically to release held descriptors
      Raise ulimit -n 65535 (see TS-HC-2) to delay symptom
- Anomaly classes: compaction
- Subsystems: TsFile, Compaction, Cache
