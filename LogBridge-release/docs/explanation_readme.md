信息收集
1. 提取“关键路径（Attention Path）”
HGT 训练时连接了最相近的 10 个窗口子图。
- 做法： 在 HGT 层中，记录 窗口节点 到 日志实例节点的注意力权重 $\alpha_{target, source}$。（仅在测试阶段收集）
- 具体细节： 步骤： 选取权重排名前3-5 条的边。
  - 物理含义： “由于窗口W与日志之间的边权重最高，说明该边是判定该异常的核心证据”。
  - 跨窗口证据： 如果权重高的边来自于“邻居窗口节点”共享的实体（例如共同访问了 Disk_01），一定要提取出来。这证明了 HGT 的有效性：它通过实体连接找到了跨窗口的冲突。
2. 提取“分类确定性（Logits Distribution）”
- 做法： 记录 Softmax 之前的原始 Logits 概率。（仅在测试阶段收集）
- 逻辑： 
  - 如果 Top-1 概率 > 80%，告诉 LLM：“模型非常确信这是 Flush 异常”。
  - 如果 Top-1 和 Top-2 概率接近（如 40% vs 38%），告诉 LLM：“模型在 Flush 和 Compaction 之间犹豫，请根据日志语义判断是否存在两者混淆的特征”。
- 具体做法：
  1. 当一个测试窗口进入模型，模型执行前向传播。
  2. 此时，通过 Hook（钩子函数）或者修改 forward 返回值（这个方法并不是固定的，你自己思考一下用什么方法好，我只是举几个例子），实时截取当前窗口对应的 HGT 注意力矩阵和最后的 Logits。
3. 提取“相似案例（Neighbor Knowledge）”
由于训练时是用实体连接找的 10 个子图，在推理时，这些子图对应的历史标签和专家注释就是最直接的 In-Context 资料。
- 做法： 获取这 10 个邻居窗口中，标签与你预测结果一致的那些样本。 
- 意义： 告诉 LLM，“在具有相似实体连接关系的窗口中，这类异常通常的根因是 X”。
- 具体做法：
  第一步：离线索引（Indexing Phase）—— 在训练完成后执行一次
  1. 冻结模型： 此时 HGT 和 Transformer 的参数已经固定。
  2. 提取训练集特征： 将你的全部训练集窗口再跑一遍模型，记录每个窗口在 HGT 输出层得到的 向量（Embedding）。
  3. 构建向量库： 
    - Key（键）： Embedding。
    - Value（值）： 这个窗口的原始日志文本、真实标签、以及你为该类异常预设的标准解释/修复建议。
  4. 持久化： 使用 Faiss 等工具将这些 Embedding 建立索引，存入磁盘。
第二步：在线检索（Retrieval Phase）—— 在测试时执行
  1. 测试窗口特征化： 得到待测窗口 $W_{test}$ 的 Embedding。
  2. 初筛： 在向量库中检索距离最近的 $Top-K$ 个样本。
  3. 二次过滤（按标签）： 获取模型对 $W_{test}$ 的预测结果（例如：Full_CPU）。
    - 从检索出的 $Top-K$ 个样本中，只选出标签也是 Full_CPU 的样本作为 LLM 的参考（但是也可以找一些很相似但异常类型不同的窗口作为反面信息，但这个我没太想好，反面信息肯定不能太多）。

解释生成
第一阶段：生成器 (The Generator) —— 初始尝试
- 输入： 1. 待测窗口日志。
1. HGT 提取的 Top-N 关键路径（线性化表示）。
2. 相似案例的专家经验（来自向量数据库检索）。
- 策略： Few-shot。提供 2-3 个“路径 -> 解释”的范例。
- 输出： 初始解释报告 $E_0$。
第二阶段：基于证据的校验器 (The Evidence-based Verifier) —— 核心创新
这一步不依赖大模型的直觉，而是利用 HGT 判别模型 提供的“硬指标”作为反馈：
1. 路径一致性检查 (Path Alignment)： * 检查 $E_0$ 中提到的异常原因是否涉及了 HGT 权重最高的那些实体或日志模板。
  - 反馈： 如果不匹配，生成反馈：“你提到的根因是 A，但模型捕获的物理证据（Path）显示冲突发生在 B，请重新评估。”
2. Logits 不确定性分析：
  - 如果 Logits 显示模型在 Class A 和 Class B 之间犹豫。
  - 反馈： “当前分类置信度较低，请重点对比 Class A 和 Class B 的细微差别。”
第三阶段：纠正器 (The Refiner) —— 迭代优化
- 大模型接收初版报告 $E_0$ + 校验器的反馈。
- 根据反馈修改逻辑漏洞，产生最终报告 $E_{final}$。
